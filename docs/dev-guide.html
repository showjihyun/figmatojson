figma-reverse — Developer Guide

Figma .fig 바이너리를 구조화된 JSON / Pencil .pen / 편집 가능한 HTML 로 무손실 변환하는 파이프라인의 기술 문서.

Technical guide for the lossless pipeline that reverse-engineers Figma .fig binaries into structured JSON, Pencil .pen, and editable HTML.

1. 개요1. Overview

figma-reverse 는 Figma 의 .fig 바이너리를 오프라인에서, API 없이, 계정 없이 무손실 분해하는 도구체인이다. 두 개의 출력 그룹과 한 개의 편집기를 동시에 제공한다:

  • 구조화 JSON + 에셋 — 사람이 읽을 수 있는 page-level JSON, SHA-256 manifest 가 동봉된 image / SVG 디렉토리
  • Pencil .pen + .pen.json — pencil.dev 호환 디자인 파일 + provenance trail
  • 편집 가능한 단일 HTML 파일 — 원본 .fig 가 base64 로 embed 된 자체 완결 dashboard
  • Web 편집기 — Konva canvas + Inspector + chat agent 가 떠있는 React 클라이언트 (web/)

핵심 가치는 round-trip 보장: .fig → extracted/ → .fig 가 byte-identical (Byte Repack) 또는 의미 동등 (Kiwi/JSON Repack) 으로 다시 만들어진다. CLI 9 단계 파이프라인 + Web 편집기가 같은 도메인 모듈 (src/) 위에 얹혀 있어 한쪽 변경이 다른 쪽에도 전파된다.

figma-reverse is an offline, no-API, no-account toolchain that losslessly decomposes Figma .fig binaries. It produces two output groups plus a web editor:

  • Structured JSON + assets — human-readable per-page JSON, image / SVG directories with SHA-256 manifests
  • Pencil .pen + .pen.json — pencil.dev-compatible design file + provenance trail
  • Single editable HTML — self-contained dashboard with the original .fig base64-embedded
  • Web editor — React client (web/) with Konva canvas, Inspector, and a chat agent

The core value claim is round-trip integrity: .fig → extracted/ → .fig reproduces byte-identical (Byte Repack) or semantically equivalent (Kiwi / JSON Repack) output. The CLI 9-stage pipeline and the Web editor sit on top of the same domain modules under src/, so a change on one side propagates to the other.

3.1 Container — ZIP 분해3.1 Container — ZIP disassembly

Figma Cloud 의 .fig 는 사실 ZIP STORE (압축 없는 ZIP). src/container.ts:loadContainer 가 ZIP magic 50 4B 03 04 또는 fig-kiwi magic 66 69 67 2D 6B 69 77 69 로 자동 분기.

A Figma .fig is actually a ZIP STORE (uncompressed ZIP). src/container.ts:loadContainer branches on ZIP magic (50 4B 03 04) vs raw fig-kiwi magic (66 69 67 2D 6B 69 77 69).

design.fig (ZIP STORE)
├── canvas.fig          ← real binary (fig-kiwi format)
├── meta.json           ← file_name, background_color, ...
├── thumbnail.png
└── images/
    ├── <sha1-hash-1>   ← raw bytes, no extension
    └── <sha1-hash-2>   ← shared by multiple nodes via image.hash

3.2 Archive + Kiwi 디코드3.2 Archive + Kiwi decode

canvas.fig 는 fig-kiwi 포맷:

canvas.fig is in the fig-kiwi format:

[8 bytes  ] "fig-kiwi" magic
[4 bytes  ] version (LE uint32)              ← metarich = 106
[4 bytes  ] chunk[0].size (LE uint32)
[N bytes  ] chunk[0].data                    ← schema chunk (deflate-raw)
[4 bytes  ] chunk[1].size
[N bytes  ] chunk[1].data                    ← data chunk (zstd!)

이중 압축이 본 프로젝트의 핵심 발견: schema chunk 는 deflate-raw, data chunk 는 zstd. fig-kiwi npm 패키지는 단일 deflate-raw 만 가정 — 우리는 magic byte 자동 감지로 분기 (src/decompress.ts). 디코드 절차:

Two-tiered compression is this project's core finding: the schema chunk is deflate-raw, the data chunk is zstd. The fig-kiwi npm package assumes a single deflate-raw — we auto-detect the algorithm by magic byte (src/decompress.ts). Decoder flow:

const archive  = parseFigArchive(canvasBytes);          // Stage 2
const schema   = kiwi.decodeBinarySchema(deflateRaw(archive.chunks[0]));
const compiled = kiwi.compileSchema(schema);            // 568 type definitions
const message  = compiled.decodeMessage(zstdInflate(archive.chunks[1]));
// → root type NODE_CHANGES { nodeChanges: NodeChange[], blobs: Bytes[], ... }

3.3 Tree — 트리 재구성3.3 Tree reconstruction

message.nodeChanges 는 평탄한 배열 — 메타리치 fixture 기준 35,660 노드. src/tree.ts:buildTree 가:

  1. 각 노드를 (sessionID:localID) 키 Map 에 저장
  2. parentIndex.guid 로 부모 link → children 배열 채움
  3. parentIndex.position (Figma 의 fractional indexing string) 으로 형제 정렬
  4. type === 'DOCUMENT' 노드를 root 로 식별

message.nodeChanges is a flat array — 35,660 nodes for the metarich fixture. src/tree.ts:buildTree:

  1. indexes each node by (sessionID:localID)
  2. links parents via parentIndex.guid → fills children
  3. sorts siblings by parentIndex.position (Figma's fractional indexing string)
  4. identifies type === 'DOCUMENT' as the root

3.4 Assets + Vectors3.4 Assets + Vectors

  • 이미지: 노드의 image.hash 가 SHA-1 ref. ZIP 의 images/<hash> 와 cross-reference. 확장자가 없으므로 magic byte sniff (PNG / JPG / WebP / GIF / SVG / PDF).
  • 벡터: VECTOR / STAR / LINE / ELLIPSE / REGULAR_POLYGON / BOOLEAN_OPERATION / ROUNDED_RECTANGLE 노드의 vectorData.vectorNetworkBlob 또는 fillGeometry[].commandsBlob 인덱스 → message.blobs[] byte → 5 개 path command (MOVE_TO / LINE_TO / CUBIC / QUAD / CLOSE) 디코드. 95% 성공률.
  • Images: image.hash on a node is a SHA-1 ref. Cross-checked against ZIP images/<hash>; magic-byte sniff for type (PNG / JPG / WebP / GIF / SVG / PDF) since extensions are missing.
  • Vectors: VECTOR / STAR / LINE / ELLIPSE / REGULAR_POLYGON / BOOLEAN_OPERATION / ROUNDED_RECTANGLE nodes carry a vectorData.vectorNetworkBlob or fillGeometry[].commandsBlob index into message.blobs[]; bytes decode to 5 path opcodes (MOVE_TO / LINE_TO / CUBIC / QUAD / CLOSE). 95% success rate.

4. Web 파이프라인4. Web pipeline

web/ 서브트리는 Hono backend (5274) + Vite/React 클라이언트 (5273) 의 동시 실행 (concurrently) 로 동작. Backend 는 sessionId 단위로 Memory + FS 에 documentJson 을 보존하고 클라이언트는 그 스냅샷을 fetch + edit + save 한다.

The web/ subtree runs Hono backend on :5274 + Vite/React client on :5273 concurrently. The backend keeps documentJson per sessionId in memory + FS; the client fetches a snapshot and patches / saves it.

5. SDD + 하네스5. SDD + Harness

Spec-Driven Development (SDD): 구현 전에 입력·출력·invariant 를 spec 으로 명문화. 코드는 spec 의 invariant 를 만족하는 한 자유. iron rule: spec 없이 merge 안 함, spec 의 invariant 와 test 의 assertion 은 1:1 대응.

Spec-Driven Development (SDD): write the input / output / invariants in a spec before implementing. Implementation is free as long as it meets the spec's invariants. Iron rule: no merge without a spec; each spec invariant maps 1:1 to a test assertion.

# --- 7 CLI subcommands (npx tsx src/cli.ts <cmd> …) ---

# 1) extract  — .fig → output/<name>/{pages,assets,…} + extracted/ breadcrumbs
npx tsx src/cli.ts extract docs/<file>.fig output/<name>
#   options: --minify  --no-document  --no-vector  --no-intermediate
#            --include-raw-message  --extracted-dir <path>

# 2) repack   — extracted/ → .fig (3 modes)
npx tsx src/cli.ts repack output/extracted/<name> out.fig --mode byte
npx tsx src/cli.ts repack output/extracted/<name> out.fig --mode kiwi
npx tsx src/cli.ts repack output/extracted/<name> out.fig --mode json

# 3) pen-export   — .fig → .pen + .pen.json (Pencil-compatible)
npx tsx src/cli.ts pen-export docs/<file>.fig --out out.pen

# 4) editable-html — .fig → single self-contained HTML with embedded .fig
npx tsx src/cli.ts editable-html docs/<file>.fig --out out.html --single-file

# 5) html-report   — extracted/ → browsable dashboard (multi-file or single)
npx tsx src/cli.ts html-report output/extracted/<name> ./dashboard
npx tsx src/cli.ts html-report output/extracted/<name> report.html --single-file

# 6) round-trip-html — extracted/ → 06_report/figma-round-trip.html viewer
npx tsx src/cli.ts round-trip-html output/extracted/<name>

# 7) tokens — .fig → tokens JSON (colors / typography / spacing)
npx tsx src/cli.ts tokens docs/<file>.fig
